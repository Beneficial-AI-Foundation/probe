# Glossary

Precise definitions of terms used across the probe ecosystem. When writing code, docs, or KB entries, use these exact terms.

## atom

The fundamental unit of analysis. An atom represents a single code definition (function, theorem, type, etc.) with its metadata: name, dependencies, source location, kind, and language. Atoms are the nodes in the call/dependency graph.

Implemented as `Atom` struct in `probe/src/types.rs`.

## code-name

A URI string that uniquely identifies an [atom](#atom). Used as dictionary keys in all probe JSON output. Format varies by language:
- **Rust**: `probe:<crate>/<version>/<module>/<Type>#<Trait><TypeParam>#<method>()`
- **Lean**: `probe:<FullyQualifiedName>`

Code-names are deterministic: same input produces same code-names. See [schema.md](schema.md#code-name-uri-format).

## stub

An [atom](#atom) representing an external dependency that was referenced but not analyzed. Identified structurally: `code-path` is `""`, `code-text` is `{0, 0}`. Stubs have `dependencies: []`. During merge, real atoms replace stubs with the same [code-name](#code-name).

See [properties.md](properties.md#p3-stub-detection-is-structural).

## envelope

The metadata wrapper around every probe JSON output file. Contains `schema`, `schema-version`, `tool`, `source` (or `inputs` for merged files), `timestamp`, and `data`. Enables type discrimination, provenance tracking, staleness detection, and schema evolution.

Implemented as `AtomEnvelope` and `MergedEnvelope<D>` in `probe/src/types.rs`. See [schema.md](schema.md#envelope).

## schema category

One of three data types that probe tools produce: **atoms** (call graph nodes), **specs** (function specifications), or **proofs** (verification results). Each category has its own merge strategy. Detected from the `schema` field in the [envelope](#envelope).

Implemented as `SchemaCategory` enum in `probe/src/types.rs`.

## merge

The composition operation that combines data from multiple probe tools into a single output. For [atoms](#atom): first-wins with [stub](#stub) replacement. For specs/proofs: last-wins. Implemented as `probe merge` subcommand.

See [schema.md](schema.md#merge-algorithm) and [properties.md](properties.md#p4-merge-associativity).

## translation mapping

A bidirectional mapping between [code-names](#code-name) across languages (e.g. Rustâ†”Lean). Generated by probe-aeneas using three-strategy matching against `functions.json`. Applied by `probe merge --translations` to add cross-language dependency edges.

See [properties.md](properties.md#p11-translation-mapping-is-1-to-1).

## SCIP

Source Code Index Protocol. A standard format for code intelligence data (definitions, references, symbols). probe-rust and probe-verus use SCIP indexes generated by rust-analyzer / verus-analyzer as the basis for call graph extraction.

## functions.json

An Aeneas-generated file mapping Rust function names to their Lean translations. Used by probe-aeneas as the source of truth for building [translation mappings](#translation-mapping). Contains entries with `lean_name`, `rust_name`, `source`, `lines`, and filtering flags.

## kind

A classification of what an [atom](#atom) represents in its language. Rust: `exec`, `proof`, `spec`. Lean: `def`, `theorem`, `abbrev`, `class`, `structure`, `inductive`, `instance`, `axiom`, `opaque`, `quot`. Determines how the atom is treated in analysis and visualization.

## extensions

Tool-specific fields on an [atom](#atom) that go beyond the core schema. Serialized as flat JSON fields alongside core fields. Must be preserved through [merge](#merge) operations. Examples: `requires-dependencies`, `type-dependencies`, `translation-name`.

## doctrine

Categorical algebra term. Each probe tool is a doctrine: it defines what [atoms](#atom) look like in its language, what [extensions](#extensions) are available, and how extraction works. The [merge](#merge) operator composes across doctrines. See `probe/docs/categorical-framework.md`.

## functor

Categorical algebra term. probe-aeneas is a functor factory: it produces the [translation mapping](#translation-mapping) that mediates composition across language boundaries. Domain knowledge about Aeneas transpilation lives in the functor; generic composition lives in [merge](#merge).

## .verilib

The output directory inside a target project where probe tools write their results. Contains `probes/` (raw tool output), `views/` (filtered projections), `translations/` (cross-language mappings), and `config.json` (user configuration).

## verus_syn

A modified fork of the `syn` Rust parser that understands Verus-specific syntax: `verus!{}` blocks, `spec fn`, `proof fn`, `requires`, `ensures`, `decreases`. Used by probe-verus for accurate parsing of Verus code. Standard `syn` cannot parse these constructs.

## spec taxonomy

A two-layer classification system in probe-verus for categorizing what each verified function proves. Layer 1: AST extraction via `CallNameCollector` (extracts function names from requires/ensures clauses). Layer 2: TOML-defined rules that match on extracted metadata and assign labels.

## Ralph Loop

The iterate-and-validate cycle for converging on correctness: implement â†’ audit â†’ fix â†’ audit â†’ fix â†’ ... â†’ validate. Each iteration uses [KB auditors](../index.md#reports) as the feedback signal. Named for the insight that single agent runs are probabilistic but rapid iteration drives quality exponentially.

## molecule

A filtered, projected view of [atoms](#atom) prepared for web UI consumption. Molecules are produced by probe-lean's `viewify` command, which filters atoms (excluding hidden, extraction artifacts, and irrelevant declarations) and reshapes them for display. Output goes to `.verilib/views/`. The term reflects that a molecule is a composed unit derived from atoms.

## view

A derived projection computed from raw probe output. Views live in `.verilib/views/` and are produced by downstream tools (e.g. probe-lean `viewify`, verilib-cli). They are not normative â€” the raw probe output in `.verilib/probes/` is the source of truth.

## scip-callgraph

A web UI application (separate from the probe tools) that visualizes call graphs, verification status, and cross-language mappings. It consumes [Schema 2.0](schema.md) JSON files produced by probe tools. Located in `baif/scip-callgraph/`.

## verilib-cli

An orchestration tool (separate from the probe tools) that coordinates running multiple probe tools on a project, manages `.verilib/` directory layout, and produces [views](#view). Located in `baif/verilib-cli/` (if present).

## Aeneas

A transpiler that translates Rust code to Lean 4 for formal verification. Aeneas generates Lean files from Rust source, along with metadata ([functions.json](#functionsjson)) mapping Rust definitions to their Lean translations. probe-aeneas uses this metadata to build cross-language [translation mappings](#translation-mapping).

## aeneas-config.yml

The project configuration file for [Aeneas](#aeneas)-transpiled repositories. probe-aeneas reads this file when invoked as `probe-aeneas extract <project_path>` to auto-detect paths: `crate.dir` specifies the Rust crate directory relative to the project root, and the project root itself is the Lean/Lake project. May also contain `aeneas_args.dest` (Lean output directory) and Charon configuration. See [probe-aeneas tool doc](../tools/probe-aeneas.md).

## Charon

A Rust compiler frontend used by [Aeneas](#aeneas) to extract structured information from Rust code. Charon produces fully qualified names for Rust functions (`rust-qualified-name`), which probe-rust can optionally include (via `--with-charon`) and probe-aeneas uses for high-confidence translation matching (Strategy 1).

## RQN

Abbreviation for `rust-qualified-name` â€” a [Charon](#charon)-derived fully qualified name for a Rust function. Used as the highest-confidence matching strategy in probe-aeneas translation generation. Example: `my_crate::module::MyStruct::method`.

## probe-extract-check

A validator tool (inside `probe/probe-extract-check/`) that checks extract JSON output against actual source code. Verifies that code-paths exist, line ranges are valid, and atom metadata is consistent with source.
