# probe-extract-check: Testing Extract JSON Correctness

Design document for verifying that JSON generated by `probe-rust extract`,
`probe-verus extract`, `probe-lean extract`, and `probe-aeneas extract`
is correct with respect to the source code they operate on.

## What "correct" means — properties to verify

| Property | Description |
|----------|-------------|
| **Completeness** | Every declaration in source has a corresponding atom |
| **No phantoms** | No atoms for declarations that don't exist in source |
| **Correct locations** | `code-path` exists, `lines-start`/`lines-end` bracket the actual declaration |
| **Correct names** | `display-name` matches the declaration at that location |
| **Correct kind** | `exec`/`proof`/`spec`/`def`/`theorem`/etc. matches the actual declaration kind |
| **Correct dependencies** | If A calls B, B is in A's dependencies; no spurious deps |
| **Referential integrity** | Every dependency target either exists as an atom key or is a known external |
| **Correct module** | `code-module` matches the actual module path |

## Three complementary testing layers

### Layer 1: Source-grounded validators (automated, per-tool)

**Status: BUILT**

A standalone validator that takes the JSON output + the source project and checks
properties against the actual source files.

**Modules:**
- `structural.rs` — envelope field validation, line range checks, referential integrity
- `source_checker.rs` — file existence, line bounds, name presence, kind matching (language-aware)
- `dep_checker.rs` — dependency name presence in caller span, located deps validation
- CLI: `probe-extract-check <json> [--project <path>] [--allow-warnings]`

**Checks performed for each atom:**
- `code-path` file exists relative to project root
- `lines-start` and `lines-end` within file bounds
- `display-name` found as identifier within the span text
- Kind matches declaration keyword (Rust `fn`/`proof fn`/`spec fn`, Lean `def`/`theorem`/`structure`/etc.)
- Lean auto-named instances tolerated (e.g., `instHasSizePoint`)

**Dependency checks:**
- Each dep's `display-name` appears in the caller's source span
- For `dependencies-with-locations`: line within caller's range, target name on that line

**Structural invariants:**
- All dependency targets exist as keys in `data` or are stubs
- `lines-start <= lines-end`, no zero values for non-stubs
- Schema envelope fields populated

### Layer 2: Golden file tests with curated micro-projects

**Status: BUILT**

Four micro-projects with hand-verified expected JSON, validated by integration tests.

| Fixture | Atoms | Edge cases covered |
|---------|-------|--------------------|
| `rust_micro` | 12 | mutual recursion, cross-module deps, no-dep functions, trait impls (disambiguated code-names), generics, closure params |
| `verus_micro` | 4 | exec/proof/spec kinds, requires/ensures deps, categorized deps (body/requires/ensures) |
| `lean_micro` | 10 | def/theorem/structure/class/instance kinds, sorry (failed verification), auto-generated instance names |
| `aeneas_micro` | 3 | translation-name mappings, paired Rust+Lean source |

**Comparison module:** `golden.rs` — structural JSON diff ignoring volatile fields (timestamp, commit, repo, tool version).

**Tests per fixture:** structural, source+deps, domain-specific assertions (kind checks, mutual recursion, categorized deps, translations, sorry status).

### Layer 3: Property-based checks

**Status: BUILT**

Checks that can run against any extract output without golden files.

**Module:** `properties.rs`

**Checks:**
- **Completeness**: regex-based declaration counting in source files vs atom count, with tolerance threshold (warns if ratio < 0.5 or > 2.0). Supports Rust and Lean.
- **Location overlap**: no two non-stub atoms share identical `(code-path, lines-start, lines-end)`.

### Live tool comparison tests

**Status: BUILT (ignored by default)**

Four `#[ignore]` integration tests that run actual extract tools against micro-projects
and compare output to golden files using the structural diff:

- `live_probe_rust_extract` — requires `probe-rust` installed
- `live_probe_verus_extract` — requires `probe-verus` installed
- `live_probe_lean_extract` — requires `probe-lean` installed
- `live_probe_aeneas_extract` — requires `probe-aeneas` installed

Run with `cargo test -p probe-extract-check -- --include-ignored` when tools are available.

## Crate structure

```
probe/
  probe-extract-check/
    Cargo.toml
    src/
      lib.rs              # Public API, check_all() orchestrator, CheckReport
      structural.rs       # Envelope fields, line ranges, referential integrity
      source_checker.rs   # File existence, name/kind matching against source
      dep_checker.rs      # Dependency name presence in caller spans
      properties.rs       # Completeness counting, location overlap detection
      golden.rs           # Structural JSON diff ignoring volatile fields
      main.rs             # CLI entry point
    tests/
      golden_tests.rs     # Integration tests (25 active + 4 ignored live tests)
      fixtures/
        rust_micro/       # 3 Rust files (lib, math, shapes) + expected.json
        verus_micro/      # 1 Verus file + expected.json
        lean_micro/       # 1 Lean file + expected.json
        aeneas_micro/     # Paired Rust+Lean + expected.json
```
